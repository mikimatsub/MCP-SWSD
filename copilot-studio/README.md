# Copilot Studio Connector Setup

This directory contains Swagger 2.0 connector specs for importing the SWSD
MCP server into Microsoft Copilot Studio. One file per profile, matching
the same profile names the server's `SWSD_PROFILE` env var accepts.

The files are regenerated by `npm run generate:swagger` (source:
[scripts/generate-copilot-swagger.ts](../scripts/generate-copilot-swagger.ts)).
Re-run after any profile change.

---

## Files

| File | Profile | Tools | When to use |
|---|---|---|---|
| `triage.swagger.yaml` | `triage` | 8 | First-line support agents who read tickets and post comments but don't reassign or close. |
| `agent.swagger.yaml` | `agent` | 22 | Full ticket-handler workflow (default). |
| `knowledge.swagger.yaml` | `knowledge` | 11 | KB authors browsing incident context plus full solution CRUD. |
| `full.swagger.yaml` | `full` | 24 | Every tool. |

The four files differ only in `info.title` and `info.description`. The
endpoint, auth, and protocol declarations are identical. Pick the file
whose label matches the `SWSD_PROFILE` your deployed MCP server is
configured with.

## Prerequisites

1. **A deployed SWSD MCP server** running with HTTP transport, reachable from
   Copilot Studio's outbound IP range. For most installs this means
   public-internet addressable (Cloud Run, Azure App Service, Render,
   etc.). For internal-only deployments, see Microsoft's docs on the
   [on-premises data gateway](https://learn.microsoft.com/en-us/data-integration/gateway/service-gateway-onprem).

   > **Don't have one yet?** Follow the [Azure Container Apps deployment recipe](../docs/deployment/azure-container-apps.md) — same Microsoft ecosystem as Copilot Studio, scale-to-zero pricing (~$0–5/month for typical team usage), public HTTPS endpoint with auto-managed TLS.

2. **An SWSD API token (JWT)** for each user who'll connect. In the
   SWSD UI: **Setup → Users & Groups → Users** → click the user → **Actions**
   → **Generate JSON Web Token**. Service Desk administrator rights are
   required to generate tokens (admins can generate on behalf of any user).
3. **Copilot Studio** with custom-connector / MCP integration enabled in
   your tenant.

## Steps

### 1. Set the host

In your chosen `*.swagger.yaml`, replace the placeholder:

```yaml
host: REPLACE_WITH_YOUR_HOST.example.com
```

with the actual host of your deployed MCP server (no protocol, no trailing
slash). Example:

```yaml
host: swsd-mcp.your-org.com
```

If your deployment uses a non-default base path, also update `basePath:`.
The default is `/` and the endpoint stays at `/mcp`.

### 2. Import into Copilot Studio

1. In Copilot Studio, open your agent.
2. Add a tool → Custom connector → New connector.
3. Choose "Import from OpenAPI file" and upload your edited `*.swagger.yaml`.
4. Step through the wizard:
   - **Host / base URL:** confirm it matches what you set in step 1.
   - **Authentication type:** API Key.
     - Parameter label: `SWSD API Token` (or anything user-friendly).
     - Parameter name: **`X-SWSD-Token`** — must match exactly.
     - Parameter location: **Header**.
   - Save the connector.

### 3. Test the connection

In Copilot Studio's connector test pane:

1. Click "Test" → "New connection".
2. Paste a valid SWSD API token.
3. Click "Test operation" on `InvokeMCP`. Provide a minimal MCP JSON-RPC
   request body for `tools/list`:

```json
{
  "jsonrpc": "2.0",
  "method": "tools/list",
  "id": 1
}
```

A successful test returns a JSON-RPC response with the tools array
containing the entries for your selected profile (e.g. 16 for `agent`).

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `401 Unauthorized` | Token not forwarded, or header name wrong | Re-check that the connector's auth header is exactly `X-SWSD-Token` (case-insensitive but spelling-exact) |
| `403 Forbidden` with "Origin not allowed" | Copilot's origin isn't whitelisted | Set `SWSD_ALLOWED_ORIGINS=` to empty (trust the proxy) or add Copilot's origin to the list |
| `400 Unsupported MCP-Protocol-Version` | Copilot sent a protocol version your server's SDK doesn't recognize | Upgrade `@modelcontextprotocol/sdk` to a version that includes the requested protocol version, or open an issue |
| Import wizard rejects the file | YAML edited incorrectly | Check `host:` line is the only thing changed; YAML indentation matters |
| Tools don't appear after successful import | Server's profile is empty or wrong | Verify `SWSD_PROFILE` env var on the server matches the Swagger file profile (or use `full`) |
