# MCP Apps Capability (Plan D) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship MCP Apps support (SEP-1865, spec 2025-11-25) on four high-value swsd tools so MCP Apps-capable hosts (Claude Web/Desktop, VS Code Copilot, ChatGPT, Goose, M365 Copilot Chat, Postman, MCPJam) render rich UI, while text-only clients (Copilot Studio authoring, generic clients) continue to receive identical structured payloads.

**Architecture:** Add `@modelcontextprotocol/ext-apps` as the UI registration layer; bundle UI sources with Vite + `vite-plugin-singlefile` into self-contained HTML files at build time; load each bundled HTML at server startup and pass it to `registerAppTool` as inline content. UI is data-driven from the same `structuredContent` payload the server already returns — no in-iframe API calls — preserving the security model and keeping clients without app capability fully functional via the existing text fallback path.

**Tech Stack:**
- `@modelcontextprotocol/sdk@^1.29.0` (bumped from `^1.26.0` to satisfy ext-apps peer)
- `@modelcontextprotocol/ext-apps@^1.7.1` (Anthropic-maintained, published 2026-04-27)
- `vite@^7.3.3` (Task 1 implementer: had to revise from `^8.0.10` — Vite 8 ships Rolldown by default, and `vite-plugin-singlefile@2.3.3` is incompatible with Vite 8's bundle output shape. Upstream PR #119 fixes this but is not yet merged. Vite 7 is actively maintained — 7.3.3 published 2026-05-07. Renovate can bump to Vite 8 after singlefile compat lands.)
- `vite-plugin-singlefile@^2.3.3`
- Plain HTML + vanilla JS for v2 UIs (no React/Svelte/Vue runtime — keeps bundles small and avoids React peer warnings)
- Existing zod 4.x, vitest 4.x, TypeScript 6.x — no changes

**XSS posture:** All UIs construct DOM with `document.createElement` + `textContent` (and `setAttribute` for known-safe attribute keys) via the safe-DOM helper module. No raw HTML strings, no template-string concatenation into HTML, no `innerHTML`-family writes. This eliminates the XSS surface even though every payload field originates from a trusted SWSD response — defense in depth.

## Strict Additivity Contract

This plan must NOT regress any existing behavior:

- Every UI-bearing tool continues to return `content: [{type: 'text', text: ...}]` plus `structuredContent: {...}` — the existing v2 response shape.
- The UI is added by attaching `_meta.ui.resourceUri` (or whatever the current ext-apps API names it — Task 1 confirms) to the existing tool registration. Tools that don't register UI are unchanged.
- The 4 UI-bearing tools' input schemas are unchanged. The `structuredContent` schemas are unchanged. The text-fallback summary strings are unchanged.
- Hosts without ext-apps support see exactly what they see today — no degradation.
- The `swsd-mcp` package's runtime dependencies remain ESM-only Node ≥ 24.15. No new build-time runtime deps (Vite is a devDependency).

A regression check (existing `tests/unit/copilotSwagger.test.ts` + e2e smoke test) must pass after every task in this plan.

Create `src/ui/shared/dom.ts`:

```ts
/**
 * Safe DOM construction helpers. No raw HTML strings.
 * Every text value goes through textContent, every attribute through
 * setAttribute. This makes XSS structurally impossible regardless of
 * payload origin.
 */

type Children = Array<Node | string | null | undefined>;
type Attrs = Record<string, string | number | boolean | undefined>;

export function el<K extends keyof HTMLElementTagNameMap>(
  tag: K,
  attrs?: Attrs,
  children?: Children,
): HTMLElementTagNameMap[K] {
  const node = document.createElement(tag);
  if (attrs) {
    for (const [key, value] of Object.entries(attrs)) {
      if (value === undefined || value === false) continue;
      if (value === true) {
        node.setAttribute(key, '');
      } else {
        node.setAttribute(key, String(value));
      }
    }
  }
  if (children) {
    for (const child of children) {
      if (child == null) continue;
      node.appendChild(typeof child === 'string' ? document.createTextNode(child) : child);
    }
  }
  return node;
}

export function clear(node: Node): void {
  while (node.firstChild) node.removeChild(node.firstChild);
}
```

- [ ] **Step 8: Create vite.config.ts**

```ts
import { defineConfig, type Plugin } from 'vite';
import { viteSingleFile } from 'vite-plugin-singlefile';
import { resolve } from 'node:path';

const UI_TOOLS = ['_smoke'];

/**
 * Flattens Vite's default multi-page HTML output (`<input-relative-path>/index.html`)
 * to `<entry-name>.html` at the outDir root. This keeps `dist/ui/<name>.html`
 * predictable for `loadUiResource(name)` while letting Rollup name JS chunks
 * however it wants.
 */
function flattenHtmlOutput(): Plugin {
  return {
    name: 'flatten-html-output',
    enforce: 'post',
    generateBundle(_options, bundle) {
      for (const [oldKey, asset] of Object.entries(bundle)) {
        if (asset.type !== 'asset' || !oldKey.endsWith('/index.html')) continue;
        const match = oldKey.match(/(?:^|\/)([^/]+)\/index\.html$/);
        if (!match) continue;
        const newKey = `${match[1]}.html`;
        delete bundle[oldKey];
        asset.fileName = newKey;
        bundle[newKey] = asset;
      }
    },
  };
}

export default defineConfig({
  plugins: [viteSingleFile(), flattenHtmlOutput()],
  build: {
    outDir: resolve(__dirname, 'dist', 'ui'),
    emptyOutDir: true,
    rollupOptions: {
      input: Object.fromEntries(
        UI_TOOLS.map((name) => [name, resolve(__dirname, 'src', 'ui', name, 'index.html')]),
      ),
    },
  },
});
```

Note: the plan's original draft used `output: { entryFileNames: '[name].html' }` to force `dist/ui/<name>.html` naming, but that collides with vite-plugin-singlefile's classifier — it sees the JS entry chunk (also named `<name>.html`) as the HTML asset and tries to inline into it. The post-build `flattenHtmlOutput` plugin runs after singlefile and renames the actual HTML asset (which Vite preserves at its original relative path under `dist/ui/`) to a flat `<name>.html` at outDir root. Future tasks add their tool name to `UI_TOOLS`.

- [ ] **Step 11: Implement the loadUiResource helper**

Create `src/mcp/uiResources.ts`:

```ts
import { readFileSync, existsSync } from 'node:fs';
import { resolve, dirname } from 'node:path';
import { fileURLToPath } from 'node:url';

const __filename = fileURLToPath(import.meta.url);
const __dirname = dirname(__filename);

// At runtime: this file is `dist/mcp/uiResources.js`, so the sibling `ui` dir
// under `dist/` is at `<__dirname>/../ui`. That's the production path.
// At test time (vitest evaluates the .ts source directly), `__dirname` points
// to `src/mcp/`, where the sibling `ui` dir contains source — not the built
// HTML — so we also try `<process.cwd()>/dist/ui` as a fallback (tests run
// from the repo root after `npm run build:ui`).
const RUNTIME_UI_DIR = resolve(__dirname, '..', 'ui');
const FALLBACK_UI_DIR = resolve(process.cwd(), 'dist', 'ui');

const cache = new Map<string, string>();

export function loadUiResource(name: string): string {
  const cached = cache.get(name);
  if (cached !== undefined) return cached;

  const candidates = [
    resolve(RUNTIME_UI_DIR, `${name}.html`),
    resolve(FALLBACK_UI_DIR, `${name}.html`),
  ];
  const path = candidates.find((p) => existsSync(p));
  if (!path) {
    throw new Error(
      `UI resource "${name}" not found at ${candidates.join(' or ')}. Did you run "npm run build:ui"?`,
    );
  }
  const html = readFileSync(path, 'utf8');
  cache.set(name, html);
  return html;
}

/** Internal — reset cache between tests. Not exported from the package. */
export function _resetCacheForTests(): void {
  cache.clear();
}
```

Note: the plan's original draft used a single `UI_DIR = resolve(__dirname, '..', 'ui')`, which works at runtime (file is at `dist/mcp/uiResources.js`) but NOT during vitest, because vitest evaluates the TypeScript source directly so `import.meta.url` resolves to `src/mcp/uiResources.ts` and `..` becomes `src/`. The dual-path fallback works in both modes without extra plumbing.

```ts
/**
 * Helpers for MCP Apps host integration. Wraps the postMessage init
 * protocol (per spec 2025-11-25 §UI Resources) so individual UIs can
 * focus on rendering.
 */

export interface HostInitMessage<T = unknown> {
  type: 'init';
  data: T;
  styles?: { variables?: Record<string, string> };
}

export function onHostInit<T>(handler: (msg: HostInitMessage<T>) => void): void {
  window.addEventListener('message', (e: MessageEvent) => {
    const msg = e.data as { type?: string };
    if (msg?.type === 'init') {
      handler(e.data as HostInitMessage<T>);
    }
  });
  // Tell the host we're ready to receive init.
  window.parent?.postMessage({ type: 'ready' }, '*');
}

export function applyHostThemeVariables(vars: Record<string, string> | undefined): void {
  if (!vars) return;
  const root = document.documentElement;
  for (const [key, value] of Object.entries(vars)) {
    root.style.setProperty(key, value);
  }
}
```

```ts
import { onHostInit, applyHostThemeVariables } from '../shared/host.js';
import { el, clear } from '../shared/dom.js';

interface Incident {
  id: number;
  number?: number;
  name: string;
  state?: string;
  priority?: string;
  assignee_email?: string;
  requester_email?: string;
  category?: string;
  updated_at?: string;
  url?: string;
}

interface ListPayload {
  incidents: Incident[];
  pagination?: { page: number; per_page: number; total?: number; total_scope?: string };
  applied_filters?: Record<string, unknown>;
}

let all: Incident[] = [];
let sortKey: keyof Incident = 'updated_at';
let sortDesc = true;

const titleEl = document.getElementById('title')!;
const searchEl = document.getElementById('search') as HTMLInputElement;
const rowsEl = document.getElementById('rows')!;
const emptyEl = document.getElementById('empty') as HTMLParagraphElement;

onHostInit<ListPayload>((msg) => {
  applyHostThemeVariables(msg.styles?.variables);
  all = msg.data.incidents;
  const total = msg.data.pagination?.total;
  const scope = msg.data.pagination?.total_scope;
  titleEl.textContent =
    total !== undefined
      ? `Incidents (${all.length} of ${total}${scope ? `, ${scope}` : ''})`
      : `Incidents (${all.length})`;
  render();
});

searchEl.addEventListener('input', render);
document.querySelectorAll('th[data-sort]').forEach((th) => {
  th.addEventListener('click', () => {
    const key = th.getAttribute('data-sort') as keyof Incident;
    if (sortKey === key) sortDesc = !sortDesc;
    else { sortKey = key; sortDesc = false; }
    render();
  });
});

function render(): void {
  const q = searchEl.value.trim().toLowerCase();
  const filtered = all.filter((i) => {
    if (!q) return true;
    return [i.name, i.assignee_email, i.requester_email, i.category]
      .filter((v): v is string => typeof v === 'string')
      .some((v) => v.toLowerCase().includes(q));
  });
  filtered.sort((a, b) => cmp(a[sortKey], b[sortKey]) * (sortDesc ? -1 : 1));

  clear(rowsEl);
  for (const i of filtered) {
    const nameCell = i.url
      ? el('td', undefined, [
          el('a', { href: i.url, target: '_blank', rel: 'noopener' }, [i.name]),
        ])
      : el('td', undefined, [i.name]);
    rowsEl.appendChild(
      el('tr', undefined, [
        el('td', undefined, [i.number !== undefined ? `#${String(i.number)}` : '']),
        nameCell,
        el('td', undefined, [i.state ?? '']),
        el('td', undefined, [i.priority ?? '']),
        el('td', undefined, [i.assignee_email ?? '']),
        el('td', undefined, [i.updated_at ? new Date(i.updated_at).toLocaleString() : '']),
      ]),
    );
  }
  emptyEl.hidden = filtered.length > 0;
}

function cmp(a: unknown, b: unknown): number {
  if (a === undefined || a === null) return 1;
  if (b === undefined || b === null) return -1;
  if (typeof a === 'number' && typeof b === 'number') return a - b;
  return String(a).localeCompare(String(b));
}
```

```ts
import { onHostInit, applyHostThemeVariables } from '../shared/host.js';
import { el, clear } from '../shared/dom.js';

interface CustomField {
  id: number;
  name: string;
  type: string;
  required: boolean;
  active: boolean;
  scope?: string;
  module?: string;
  values?: string[];
  help_text?: string;
  searchable: boolean;
}

interface CustomFieldsPayload {
  custom_fields: CustomField[];
}

let all: CustomField[] = [];

const titleEl = document.getElementById('title')!;
const searchEl = document.getElementById('search') as HTMLInputElement;
const scopeFilter = document.getElementById('scope-filter') as HTMLSelectElement;
const moduleFilter = document.getElementById('module-filter') as HTMLSelectElement;
const activeOnly = document.getElementById('active-only') as HTMLInputElement;
const rowsEl = document.getElementById('rows')!;
const emptyEl = document.getElementById('empty') as HTMLParagraphElement;

onHostInit<CustomFieldsPayload>((msg) => {
  applyHostThemeVariables(msg.styles?.variables);
  all = msg.data.custom_fields;
  titleEl.textContent = `Custom Fields (${all.length})`;
  populateFilter(scopeFilter, all.map((f) => f.scope).filter((s): s is string => !!s));
  populateFilter(moduleFilter, all.map((f) => f.module).filter((s): s is string => !!s));
  render();
});

[searchEl, scopeFilter, moduleFilter, activeOnly].forEach((eli) =>
  eli.addEventListener('input', render),
);

function populateFilter(select: HTMLSelectElement, values: string[]): void {
  const unique = Array.from(new Set(values)).sort();
  for (const v of unique) {
    select.appendChild(el('option', { value: v }, [v]));
  }
}

function render(): void {
  const q = searchEl.value.trim().toLowerCase();
  const scope = scopeFilter.value;
  const mod = moduleFilter.value;
  const onlyActive = activeOnly.checked;
  const filtered = all.filter((f) => {
    if (onlyActive && !f.active) return false;
    if (scope && f.scope !== scope) return false;
    if (mod && f.module !== mod) return false;
    if (!q) return true;
    return (
      f.name.toLowerCase().includes(q) ||
      (f.help_text?.toLowerCase().includes(q) ?? false)
    );
  });
  clear(rowsEl);
  for (const f of filtered) rowsEl.appendChild(renderCard(f));
  emptyEl.hidden = filtered.length > 0;
}

function renderCard(f: CustomField): HTMLLIElement {
  const badges = el('div', { class: 'badges' });
  if (f.required) badges.appendChild(el('span', { class: 'badge required' }, ['Required']));
  if (!f.active) badges.appendChild(el('span', { class: 'badge inactive' }, ['Inactive']));
  if (f.searchable) badges.appendChild(el('span', { class: 'badge' }, ['Searchable']));
  if (f.scope) badges.appendChild(el('span', { class: 'badge' }, [f.scope]));
  if (f.module) badges.appendChild(el('span', { class: 'badge' }, [f.module]));

  const card = el('li', { class: 'card' }, [
    el('header', undefined, [
      el('h2', undefined, [f.name]),
      el('span', { class: 'type' }, [f.type]),
    ]),
    badges,
  ]);
  if (f.help_text) card.appendChild(el('p', { class: 'help' }, [f.help_text]));
  if (f.values?.length) {
    const list = el('ul');
    for (const v of f.values) list.appendChild(el('li', undefined, [v]));
    card.appendChild(
      el('details', undefined, [
        el('summary', undefined, [`${f.values.length} values`]),
        list,
      ]),
    );
  }
  return card;
}
```

## Open questions for the implementer

These are deliberately not answered in the plan because the answer depends on what Task 1 step 1 finds in the ext-apps README — the implementer should resolve them before proceeding past Task 1:

1. **What is the actual exported registration function?** The plan calls it `registerAppTool` per the proposal. If ext-apps@1.7.1 names it differently, every Task 2-5 step that says "wire `registerAppTool`" needs the actual name.

2. **How is the UI HTML attached to the tool?** Inline string vs. resource URI vs. some other mechanism. The plan assumes inline-string (`ui: { html: <string> }`); if ext-apps requires a resource-URI registration step, Task 1's helper module needs to also register the UI as a resource and Task 2-5 wire by URI instead.

3. **Is there a separate capability declaration step?** If `McpServer({ capabilities: { experimental: { app: true } } })` (or similar) is required, add it in Task 6 step 5.

4. **How does the host inject data into the iframe?** The plan assumes a `postMessage` `init` event carrying `{type, data, styles}`. If ext-apps ships a higher-level client helper, prefer it over the hand-rolled `src/ui/shared/host.ts` and update accordingly.

These are intentionally questions, not assumptions — the implementer should not pretend to know.
