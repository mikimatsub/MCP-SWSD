# SolarWinds Service Desk MCP Server — v0.0 Validation Runbook

**Companion to:** `swsd-mcp-implementation-planning-document.md`  
**Audience:** The agent or operator executing v0.0 of the SWSD MCP server project.  
**Environment:** Operator's production SWSD tenant. Admin token. Single-tenant scope.  
**Goal:** Resolve every "Pending live" and "Not validated" entry in the planning document's API validation matrix by hitting real endpoints, recording fixtures, and capturing tenant-specific behavior.  
**Output:** A repo subdirectory `validation/` containing recorded fixtures, a filled validation matrix, a custom-field schema reference, and a tenant-notes document.

This runbook is procedural. Execute the sections in order. Each section has a goal, a procedure, an expected outcome, and what to commit. Stop and surface findings if any step produces unexpected behavior — do not "work around" anomalies during validation, capture them.

---

## 0. Pre-flight

### 0.1 Confirm working environment

- Confirm operator has a current SWSD admin JWT.
- Confirm the tenant's API base URL (`https://api.samanage.com` for US, `https://apieu.samanage.com` for EU). Record which one applies.
- Confirm operator's email and SWSD user ID — needed for assignee/requester fields.
- Set environment variables in the validation shell:

```bash
export SWSD_TOKEN="<admin JWT>"
export SWSD_BASE_URL="https://api.samanage.com"
export SWSD_TEST_MARKER="[MCP-VAL]"
export SWSD_OPERATOR_EMAIL="<operator email>"
```

### 0.2 Test-data conventions

Production validation requires guardrails. Adopt all of these:

- **Subject marker.** Every test incident, comment, and solution title begins with `[MCP-VAL]` so they're trivially filterable and removable.
- **Requester and assignee.** Every test ticket has the operator as both requester and assignee unless a specific test requires otherwise. Never assign to a real coworker during validation.
- **Test category.** Create a category named `MCP-Validation` in SWSD (Setup → Categorization → Categories) before starting. All test tickets use this category. This makes cleanup trivial and isolates test data from production reporting.
- **No notifications fan-out.** Disable email notifications on the operator's user for the duration of validation, OR confirm the test category does not have notification rules attached. Either approach is acceptable; document which was chosen.
- **SLA awareness.** If the tenant has SLA timers attached to incident states or categories, note this. State-transition probes should target tickets in the test category to avoid tripping production SLAs.
- **Workflow rule awareness.** If the tenant has workflow rules that fire on state changes, assignment changes, or category changes, note this. Validation may inadvertently trigger them — that's why the test category exists.

### 0.3 Cleanup discipline

- Maintain a list of every incident ID, comment ID, and solution ID created during validation in `validation/created-artifacts.json`.
- Plan to either close-with-resolution-note or hard-delete every test artifact at the end of v0.0. Operator picks the approach in section 9 of this runbook.
- If validation is interrupted, the partial artifacts list still allows cleanup later. Commit the file as you go.

### 0.4 Fixture scrubbing

- Every captured response fixture is committed to the repo and visible to coworkers and (eventually) the public via GitHub.
- Before commit, scrub: tokens, real user emails (replace with `user1@example.com`, `user2@example.com`), real ticket descriptions that may contain sensitive content, IP addresses, hostnames, and any custom-field values that look like business data.
- Keep response shape and field names intact. The fixture's purpose is shape, not content.
- A scrubbing helper script (`scripts/scrub-fixture.ts`) should be among the first artifacts produced — write it before recording any fixtures.

## 1. Header negotiation probe

### Goal

Lock the canonical Accept header version for this tenant and document any response-shape differences between `v1.1` and `v2.1`.

### Procedure

```bash
# Probe v1.1
curl -sS -i \
  -H "X-Samanage-Authorization: Bearer $SWSD_TOKEN" \
  -H "Accept: application/vnd.samanage.v1.1+json" \
  "$SWSD_BASE_URL/incidents.json?per_page=1" \
  > validation/headers/v1.1-response.txt

# Probe v2.1
curl -sS -i \
  -H "X-Samanage-Authorization: Bearer $SWSD_TOKEN" \
  -H "Accept: application/vnd.samanage.v2.1+json" \
  "$SWSD_BASE_URL/incidents.json?per_page=1" \
  > validation/headers/v2.1-response.txt

# Probe with no version (some integrations omit it)
curl -sS -i \
  -H "X-Samanage-Authorization: Bearer $SWSD_TOKEN" \
  -H "Accept: application/json" \
  "$SWSD_BASE_URL/incidents.json?per_page=1" \
  > validation/headers/no-version-response.txt
```

### Expected outcome

- All three return 200 OR one returns 406 / 415. Record which.
- Compare body structures. Note any field differences (renames, additions, type changes).
- Compare headers — particularly `X-Total-Count`, `Link`, rate-limit-related headers.

### Commit

- `validation/headers/*.txt` (scrubbed)
- `validation/headers/findings.md` documenting the canonical version and any differences. Update planning doc section 2.3 if needed.

## 2. Lookup endpoint discovery

### Goal

Capture the response shape of every read-only lookup endpoint the four profiles depend on.

### Procedure

For each of these endpoints, GET with `per_page=5`, record the response, scrub, commit to fixtures:

```text
/categories.json
/sites.json
/departments.json
/users.json
/groups.json
/roles.json
/priorities.json        (may not exist — try and record)
/states.json            (may not exist — try and record)
```

Use the standard SWSD headers (canonical Accept version from section 1).

```bash
for endpoint in categories sites departments users groups roles priorities states; do
  curl -sS \
    -H "X-Samanage-Authorization: Bearer $SWSD_TOKEN" \
    -H "Accept: application/vnd.samanage.v2.1+json" \
    "$SWSD_BASE_URL/${endpoint}.json?per_page=5" \
    > validation/lookups/${endpoint}.json
done
```

### Expected outcome

- Most endpoints return 200 with arrays.
- Some may 404 — that tells you the resource doesn't exist as a top-level endpoint and priorities/states are likely enumerated values discovered via incident responses instead.
- Pagination headers should be present on list responses — capture `X-Total-Count` and any `Link` header values.

### Commit

- `validation/lookups/*.json` (scrubbed)
- `validation/lookups/findings.md` documenting which endpoints exist, their response shape, and pagination behavior. Note any endpoints that return 404 — these need alternative implementation strategies in the planning doc tool catalog.

```bash
  "$SWSD_BASE_URL/incidents.json?per_page=5&updated[]=greater_than&updated[]=2026-04-01" \

# 3.4 Get single incident — pick an ID from the list response
INCIDENT_ID="<id from 3.1>"
curl -sS \

  -H "Accept: application/vnd.samanage.v2.1+json" \

# 3.5 Pagination probe — confirm X-Total-Count and how next pages work
```

- Single incident response contains `custom_fields_values` array — capture its shape carefully, this drives section 6.
- Filter syntax confirmed (or revealed to be different from the brackets-array convention).
- `X-Total-Count` header present, `Link` header may or may not be present.
- Per-page cap discovered (community examples suggest 100; verify).

### Commit

- All `validation/incidents/*.json` (scrubbed)
- `validation/incidents/findings.md` documenting filter syntax, pagination, max per_page, and the response shape of `custom_fields_values`.

## 6. Custom-field schema introspection

### Goal

Capture the operator's tenant's complete custom-field configuration as a reference fixture. This is the single most important deliverable of v0.0 — it unblocks the whole custom-fields strategy.

### Procedure

There may not be a single canonical metadata endpoint. Try in order:

```bash
# 6.1 Try /custom_fields.json
curl -sS -i \
  -H "X-Samanage-Authorization: Bearer $SWSD_TOKEN" \
  -H "Accept: application/vnd.samanage.v2.1+json" \
  "$SWSD_BASE_URL/custom_fields.json" \
  > validation/custom-fields/endpoint-probe-1.txt

# 6.2 Try /custom_forms.json
curl -sS -i \
  -H "X-Samanage-Authorization: Bearer $SWSD_TOKEN" \
  -H "Accept: application/vnd.samanage.v2.1+json" \
  "$SWSD_BASE_URL/custom_forms.json" \
  > validation/custom-fields/endpoint-probe-2.txt

# 6.3 Try /catalog_items.json
curl -sS -i \
  -H "X-Samanage-Authorization: Bearer $SWSD_TOKEN" \
  -H "Accept: application/vnd.samanage.v2.1+json" \
  "$SWSD_BASE_URL/catalog_items.json" \
  > validation/custom-fields/endpoint-probe-3.txt
```

### Fallback if no metadata endpoint exists

If none of the probes return useful schema metadata:

1. Pull a representative sample of incidents (10–20 across multiple categories) to capture the union of `custom_fields_values` shapes that appear in practice.
2. Pull a representative sample of solutions if solutions also support custom fields.
3. Manually inventory the custom fields via the SWSD admin UI (Setup → Custom Fields) and transcribe into a JSON document keyed by field name with type, scope, required-ness, and any allowed values for dropdowns.
4. Save the inventory as `validation/custom-fields/tenant-schema.json`.

### Commit

- `validation/custom-fields/endpoint-probe-*.txt`
- `validation/custom-fields/tenant-schema.json` — the canonical reference for `swsd_describe_custom_fields`.
- `validation/custom-fields/findings.md` documenting the introspection method that worked, any quirks (overnight indexing, scoped fields, etc.), and how writes target fields (by name vs by id).

## 7. Write endpoint validation

### Goal

Confirm every write tool's exact endpoint, method (PUT vs PATCH), payload shape, and error behavior.

### Procedure

```bash
curl -sS -X POST \
  -H "X-Samanage-Authorization: Bearer $SWSD_TOKEN" \
  -H "Accept: application/vnd.samanage.v2.1+json" \
  -H "Content-Type: application/json" \

    "incident": {
      "name": "[MCP-VAL] minimal create test",
```

## 9. Cleanup

### Goal

Leave production with no test data.

### Procedure

For each artifact ID in `validation/created-artifacts.json`:

#### Option A — close-with-resolution (preferred for incidents)

```bash
curl -sS -X PUT \
  -H "X-Samanage-Authorization: Bearer $SWSD_TOKEN" \
  -H "Accept: application/vnd.samanage.v2.1+json" \
  -H "Content-Type: application/json" \
  -d '{ "incident": { "state": "Resolved", "solution": "Closed by MCP validation cleanup" } }' \
  "$SWSD_BASE_URL/incidents/${ID}.json"
```

#### Option B — hard delete (preferred for solutions, comments, and if operator wants no trace)

```bash
curl -sS -X DELETE \
  -H "X-Samanage-Authorization: Bearer $SWSD_TOKEN" \
  -H "Accept: application/vnd.samanage.v2.1+json" \
  "$SWSD_BASE_URL/incidents/${ID}.json"
```

After cleanup, run a final search:

```bash
curl -sS \
  -H "X-Samanage-Authorization: Bearer $SWSD_TOKEN" \
  -H "Accept: application/vnd.samanage.v2.1+json" \
  "$SWSD_BASE_URL/incidents.json?name=%5BMCP-VAL%5D&per_page=50"
```

Confirm zero open `[MCP-VAL]` incidents remain. Same for solutions.

### Commit

- `validation/cleanup-log.md` documenting which option was used for each artifact and confirming the final search returned empty.

## Notes for the executor

- **Don't optimize during validation.** If a probe returns something unexpected, capture it and move on. The point is to discover the API as it actually behaves, not to make it conform to expectations.
- **Don't skip findings docs.** The fixtures alone aren't enough — the `findings.md` files are what future-you reads when implementing the tools. Write them while the context is fresh.
- **Surface anomalies up.** If something behaves so oddly that the planning doc's tool catalog needs revision, stop and surface the finding rather than silently re-architecting.
- **Cleanup is non-negotiable.** Production must be left clean. The cleanup-log.md is a hard exit criterion.
